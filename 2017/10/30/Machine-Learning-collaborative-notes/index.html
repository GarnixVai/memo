Data Models

  • 主概念
    • a collection of tables, termed relations( each with a unique name.)
    • a relation: tables with rows and columns
    • 每個relation都有schema(for rows and columns)
    • mathematical concept: sets
  • 名詞連結
    • column, –> attribute
    • value, correspond to an attribute or a field
    • row(tuple/ record), –> a collection of related values
  • 數學關係 part1 (domain)
    • a relation(table) : a subset of domain
    • a relation with attributes (A1:D1, A2:D2, …, An:Dn)–> Domain: D1 X D2 X…Dn
    • Example(p.11)
  • 數學關係 part2 (database)
    • Database schema
      • logical design
    • Database instance
      • a set of memory structures that manage database files.
      • The data in the database at a given instant in time(what does this mean??) 在那當下資料庫裡的資料？
  • 表示方法
    • binary search tree
      • based on primary key
    • array ( vector)
      • primary key as array index
    • hash table
      • hash primary key
    • linked list
      • tuples
  • 創立relation
    • 等同於創建table
      • SQL CREATE TABLE table_name (A1 D1, A2 D2, …, An Dn);
      • A: attribute name
      • D: domain data type
  • relation 特色
    • tuples be ordered
      • Attribute:value = R(A1, A2, …, An):t (oredered)
    • all values atomic(indivisible)

Relational Integrity Constraints

  • 種類
    • Domain constraints.
      • 每個value都該為那attr設定的單元值
      • EX:若為Interger就不該出現小數
    • Key constraints
      • 每個relation(table)中的tuples值都該為唯一
  • 實體完整性限制( Entity Integrity Constraints)
    • Primary key can’t be null
    • t[PK] not null for any tuple t in r( R ) P18

  • 探討

    • 多個attribute name 可為相同concept ，也可為不同 concept

  • 參考完整性限制(Referential Integrity Constraints)

    • 關聯模式裡最重要的一種限制
    • 序列值裡的foreign key，可以是Null
    • 序列值裡的foreign key若非Null，則該值必須存在於其所參考的關聯之主鍵值裡。
    • 維繫不同relation中的tuples的關鍵
  • 他種限制 :語意完整限制(Semantic integrity constraint)
    • 取決於應用層P22

Normalization

  • 導論
    • attr of diff entities 不應用同樣的relation 表示
    • 只有foreign key 該被指向其他entities
    • entity 和 relationship attr 應要區分清楚
    • 不當的attr和 entity設計可能造成問題
      • 浪費儲存空間
      • 在進行資料庫動昨時造成異常(Anomalies)
    • 異常
      • Insert
      • Delete
      • Modify (P27)
    • 設計schema時避免往後容易出現的異常
    • 設計時盡量避免 tuples 出現null值(null的三大可能性: invalid、unkown(but exist), unavailable)
    • Lossless join condition: 名詞解釋, Wiki
      • Ex: R(A,B,C,D)
        FD: F={A->B, B->C, C->D}
        R1(A,D), R2(B,D), R3(C,D) : not lossless
        R1(A,B), R2(A,C), R3(A,D) : lossless

  • Functional dependencies

    • 判斷資料庫設計好壞的方法
      • X–>Y
      • if t1[X] = t2[X], then t1[Y] = t2[Y]
      • X–> Y in R : 代表所有r( R ) 都具有該項限制

  • Inference Rules

    • Armstrong’s inference rules:
      • Reflexive: Y subset of X then X–> Y
      • Augmentation: X–> Y, then XZ–>YZ
      • Transitive: X–>Y
  • Equivalence of Sets of FDs(P.33 done later)
  • Normalization of Relations
    • break attr into smaller relations
  • Normal Form ref
    • 4 forms: use keys and FD to identify
    • Targets: high quality design and properties
    • Problems: some are hard to understand/detect
    • Denormalization:
  • First Normal Form
    • having PK, 欄位相依
    • 每個欄位只能有單一值(不可重複、多個)
    • 不允許意義相同的多個欄位
  • Second Normal Form
    • 符合1NF
    • 各欄位與PK間不存在部分相依的關係
      • 分割『欄位一在重複』的欄位 分割『欄位一在重複』的欄位
  • 3 NF
    • 符合2NF
    • 除去間接相依的關係
      • 分割『與主鍵無關』的欄位 分割『與主鍵無關』的欄位
  • BCNF(老師上課時說這個不太重要，所以我不多做XDDDDD)
  • 4NF
    • 符合3NF
    • 概念不同的元素完全分開

  • 5NF

    • 符合4NF
    • 分割至最小的可能

Ch5 Storage and Indexing Handout

  • Q: Use examples to describe the advantages/disadvantages of creating an index in
    DBMS

  • Record id(Rid), 可用 indexes 快速找

  • Buffer Manager, 把 disk page read 進 main memory
  • Index Classification
    • Search key 含有 primary index -> primary index, 其餘 secondary index
    • Unique index -> 含有 candidate key (candidate key是啥?)
    • Data entry(index的部分) Data Record(真實的資料)
    • Clustered index, Data entry & Data Record排序相同or相近
    • 頂多只能用一個search key做clustered index
  • Order m Btree
    • node最多m個children
    • node最少$\lceil m/2 \rceil$個children
    • insert, delete
      • https://hackmd.io/CwdgzAbAphDGYFoCGBOAHI4UBGIFoEYCAGfMA4YgJiWJFmIFYg==
      • 講義

  • Hashing m buckets

    • depth，bucket數量總共有$2^d$個，d為depth(hash過後的數字會比d還多位，但只用後面數來d位做bucket的編號)
    • Collision
      • insert新進資料的bucket已滿
    • 靜態hashing的缺點
      • 固定的m個bucket不好，資料庫可能有增減
      • 有序性的access效率不佳
    • Overflow的處理
      • overflow page(頭痛醫頭，腳痛醫腳，發生問題才解決)
      • double the size of buckets(不太可能，硬體限制page大小固定)
      • double the number of buckets and reorganize(reorganize 全部很貴，reorganize一個就OK，所以產生extendible hashing)
    • Extendible hashing
      • Double bucket when collision, 但只reorganize overflow的那個
    • Linear hashing

  • 選擇index時，考慮後果。query變快，update變慢，占用disk.

  • Range query -> tree
  • Exact match -> hash (==)
  • Raid
    • Redundant Arrays of Inexpensive Disks

Ch7 ER Model Handout

  • 基礎觀念
    • 以物件組成的系統
    • 物件之間的關係
  • Attribute Types
    • Composite(上下組成關係) vs. Simple(單一單位)
    • 多值 vs 單一值
    • 可衍生(derived)　vs Stored(??What)
    • Null
  • E-R: Relational sets
    • Relationship: 連結兩個以上的entity
    • Relationship set: 多個relation
      • relationship set : 菱形(動詞/介係詞)
      • entity: 長方形(名詞)
      • attr: 橢圓形(包含的性質/特性(名詞))
    • 一個entity可扮演多種角色
  • Mapping Cardinalities
    • 1:1–>man marry womam
    • N:1–>children have a mom
    • 1:N–>mother have children
    • M:N–>student enroll classes
  • Key
    • 讓entity 和 relationship更加特性分明(=3=)
    • Superkey: identify entity(但可由多個PK組成)
    • Candidate key: 較小的superkey(?)
      • Can be any column or a combination of columns that can qualify as unique key .
      • Can be multiple in one table.
      • Each Candidate Key can qualify as Primary Key.
    • Primary key : identify entity
  • Entity
    • Weak Entity: 沒有能成為PK的attr
      • 必須依靠其他entity才能存在
      • 例如1:N的關係
    • Strong Entity: 有PK
    • High-leval Entity: 非常好用，用來連結多個系統、過渡元素的超強模組(自我解釋)
